# PhonePe Payment + Order Placement Flow (Current Codebase)

This document captures the exact PhonePe flow currently implemented in this app so it can be reused in a new codebase with minimal ambiguity.

Primary source files:
- `lib/presentation/cart/cart_page.dart`
- `lib/presentation/cart/cart_controller.dart`
- `lib/infrastructure/network/dio_client.dart`
- `lib/infrastructure/network/api_constants.dart`
- `lib/infrastructure/constants/app_constants.dart`

---

## 1) Functional Goal

In cart checkout:
- user taps PhonePe,
- app creates a pending order first,
- payment is processed via PhonePe SDK,
- payment status is verified,
- pending order is marked paid and confirmed.

This ensures order is finalized only after payment success.

---

## 2) High-Level Architecture

The flow is a **two-phase commit** style process:

1. **Phase A: Pending Order Creation**
   - backend order is created with `payment_status = pending`
   - returned `order_id` is stored

2. **Phase B: Payment Confirmation**
   - PhonePe SDK transaction is launched
   - app checks PhonePe status endpoint
   - on success: backend payment success API marks pending order as paid
   - on failure/cancel: backend payment failure API is called

---

## 3) User Entry Point in UI

In `cart_page.dart`, PhonePe button tap:
- validates user login
- calls:

`await controller.placeOrderWithoutPayment('phonePe');`

So PhonePe always starts from `placeOrderWithoutPayment`.

---

## 4) End-to-End Sequence (Current Implementation)

1. User taps PhonePe on cart screen.
2. `placeOrderWithoutPayment('phonePe')` executes.
3. App prepares PhonePe transaction id/checksum/body.
4. App calls backend to create order with `payment_status = pending`.
5. Backend returns `order_id`; app stores in `newOrderId`.
6. App launches PhonePe SDK with `startTransaction(...)`.
7. On SDK callback, app calls PhonePe status API.
8. If status = completed:
   - app calls backend payment success API with `order_id` + transaction details.
9. If status failed/cancelled:
   - app calls backend payment failure API.
   - shows payment failed bottom sheet.
   - performs cleanup call for pending order.
10. Success path clears cart id and navigates to order success.

## 5) Core Methods and Responsibilities

## `placeOrderWithoutPayment(paymentMethodType)`
Location: `cart_controller.dart`

Responsibilities:
- logs begin-checkout analytics
- calls `getPhonePeCheckSum()` to prepare PhonePe transaction metadata
- creates pending backend order
- stores backend `order_id` in `newOrderId`
- branches:
  - `phonePe` -> `startPgTransaction()`
  - otherwise -> Razorpay flow

Key outgoing payload fields:
- `restro_id`
- `cart_id`
- `price`
- `total_paid`
- `pickup_date`
- `pickup_time`
- `pickup_end_time`
- `gst_charge`
- `payment_type = ONLINE`
- `payment_status = pending`
- `payment_method = phonePe`
- `transaction_amount`
- `transaction_payment_type = online`
- `payment_id` (merchant txn id generated by app)

Notes:
- Optional phone override:
  - `country_code`
  - `mobile`

---

## `getPhonePeCheckSum()`
Location: `cart_controller.dart`

Responsibilities:
- generates merchant transaction id (`phonePeTId`)
- builds PhonePe request body
- base64 encodes body
- computes checksum:
  - `sha256(base64Body + "/pg/v1/pay" + saltKey) + "###" + saltIndex`

PhonePe request body fields:
- `merchantId`
- `merchantTransactionId`
- `merchantUserId`
- `amount` (paise, i.e. INR * 100)
- `callbackUrl`
- `redirectUrl`
- `mobileNumber`
- `paymentInstrument.type = PAY_PAGE`

---

## `startPgTransaction()`
Location: `cart_controller.dart`

Responsibilities:
- initializes PhonePe SDK (`phonePeInit()`)
- calls `PhonePePaymentSdk.startTransaction(...)`
- forwards response to `phonePeTransactionResponse(...)`

Current invocation parameters:
- body: base64 payload
- callback URL
- checksum
- package name: `com.good.grab`

---

## `checkPhonePeStatusApi()`
Location: `cart_controller.dart`

Responsibilities:
- validates transaction from PhonePe status endpoint
- computes status `X-VERIFY`:
  - `sha256("/pg/v1/status/{merchantId}/{merchantTxnId}" + saltKey) + "###" + saltIndex`
- checks success condition:
  - `success == true`
  - `code == PAYMENT_SUCCESS`
  - `data.state == COMPLETED`

Branching:
- success -> `placeOrderWithPayment(transactionId, 'phonePe')`
- failure/cancel ->
  - `_reportPaymentFailure(...)`
  - show `PaymentFailedWidget`
  - `removeOrderData()`

---

## `placeOrderWithPayment(transId, paymentMethodType)`
Location: `cart_controller.dart`

Responsibilities:
- calls backend payment success endpoint
- binds successful payment to the pending order
- clears cart id
- navigates to order success screen with order metadata

Current success payload:
- `order_id` (pending order id)
- `payment_status = paid`
- `original_transaction_id` (PhonePe txn id)
- `cart_id`
- `payment_id` (merchant txn id)
- `payment_type = ONLINE`
- `transaction_status = success`
- `payment_method = phonePe`
- `transaction_payment_type = online`

Expected backend behavior:
- mark pending order as paid
- return normalized order success response including `order_id`

---

## `_reportPaymentFailure(...)`
Location: `cart_controller.dart`

Responsibilities:
- calls backend payment failure endpoint with minimal data:
  - `order_id`
  - `payment_status` (`Failed` or `Cancelled`)

Used when payment is not completed.

---

## `removeOrderData()`
Location: `cart_controller.dart`

Responsibilities:
- triggers cleanup endpoint call with:
  - `order_id`

Current implementation calls `funPlaceCompleteOrderApi` for cleanup semantics.

## 6) API Mapping in Network Layer

From `api_constants.dart` + `dio_client.dart`:

- Pending order creation:
  - method: `funPlaceOrderWithoutPaymentApi`

  - method: `funPlaceOrderPaymentApi`
  - endpoint: `apiPaymentSuccess` (`/order-payment-success`)

- Payment failure update:
  - method: `funPlaceOrderPaymentFailApi`
  - endpoint: `apiPaymentFail` (`/order-payment-failed`)

- Alternate/legacy completion call:
  - method: `funPlaceCompleteOrderApi`
  - endpoint: `apiPlaceOrderWithTransaction` (`/place-order-with-transaction`)

## 9) Recommended Implementation in New Codebase

Use this exact control flow:

1. Validate login + phone.
2. Generate merchant txn id.
3. Create pending order in backend.
4. Persist pending `order_id` in memory/session.
5. Start PhonePe SDK.
6. Verify payment status from PhonePe status API.
7. If completed:
   - call payment success API against pending order.
8. If failed/cancelled:
   - call payment failed API.
   - keep UX retry-friendly.
9. Clear cart only after success API returns success.

Critical rule:
- **Never finalize order before payment success verification.**

## 10) Production Hardening Notes (Important)

These are visible in current code and should be improved when porting:

1. Amount calculation:
   - `totalPay` is currently set to `1` in cart methods (test behavior).
   - Replace with real payable amount before production rollout.

2. Secrets in app:
   - `phonePeSaltKey` and other sensitive keys are in client constants.
   - move signature generation and secret handling to backend where possible.

3. Callback URLs:
   - payload currently hardcodes callback URL.
   - centralize via backend config/env to avoid mismatch.

4. Payment method visibility:
   - backend flags `phonepe`/`razorpay` are fetched but not used in UI gating.
   - conditionally render available methods from backend capabilities.

5. Idempotency:
   - ensure payment success endpoint is idempotent by `order_id` + transaction id.
   - protects from duplicate callback/status checks.

6. Retry strategy:
   - add bounded retry for status checks and temporary network failures.

## 11) Suggested Backend Requirements for Clean Port

Backend should support:
- pending order creation with immutable `order_id`
- payment success confirmation that transitions order state atomically
- payment failure endpoint for cancellation/failure bookkeeping
- idempotent updates
- strict validation:
  - amount consistency
  - merchant transaction id mapping
  - order ownership/user ownership

## 12) Minimal Pseudocode for New Project

```dart
Future<void> checkoutWithPhonePe() async {
  ensureUserAndPhone();

  final merchantTxnId = generateTxnId();
  final pendingOrderId = await api.createPendingOrder(
    paymentMethod: 'phonePe',
    paymentId: merchantTxnId,
    paymentStatus: 'pending',
  );

  final sdkPayload = buildPhonePePayload(merchantTxnId, amountInPaise);
  await phonePeSdk.startTransaction(sdkPayload);

  final status = await phonePeApi.checkStatus(merchantTxnId);
  if (status.isCompleted) {
    await api.markPaymentSuccess(
      orderId: pendingOrderId,
      originalTxnId: status.transactionId,
      paymentId: merchantTxnId,
    );
    clearCart();
    navigateSuccess();
  } else {
    await api.markPaymentFailed(orderId: pendingOrderId, status: status.reason);
    showPaymentFailed();
  }
}
```

## 13) Quick QA Test Matrix

Validate these scenarios in new codebase:

1. PhonePe success -> order success persisted and cart cleared.
2. User cancels at PhonePe -> order not confirmed, failure endpoint called.
3. Payment fails -> failure endpoint called and retry possible.
4. App kill/reopen after pending order -> recovery path handles incomplete transaction.
5. Duplicate success callback/status check -> no duplicate confirmed orders.
